## v2 CuTe-DSL kernel (5 steps, bit-exact)

### Step 5 (`ec85630`): smem-staged epilogue
Replaced direct register→gmem store with proper staged epilogue:
`registers (fp32 acc) → sC (bf16) → gmem (bf16)`.  Uses
`make_tiled_copy_C_atom(copy_atom, tiled_mma)` for the r2s retile.
Bit-exact on every M=16 Nano35 lock-in shape (cos=1.0).

### Step 4 (`a38d118`): full K-loop MMA
Implemented complete K-loop with MMA accumulation.  Tiled MMA:
`MmaF16BF16Op(16, 8, 16)`, `atom_layout=(1, 4, 1)` (4 warps along N),
`permutation_mnk=(16, 32, 16)`.  Block sizes adjusted:
BLOCK_M=16, BLOCK_N=32 (was 64), BLOCK_K=32.  Bit-exact
(max_abs=0.0, cos=1.0) on (m=16, k=128, n=32) probe.

`is_supported` tightened to require `m == BLOCK_M` because the v1
kernel writes a full tile per CTA with no residual handling.

### Step 3 (`7eb7763`): FP4 nibble dequant — bit-exact
Per-thread FP4 nibble decode in registers, multiplied by per-block
FP8 scale (loaded fp32 from gmem) and scalar alpha, bf16-cast and
stored to sB.

Caught two bugs:
1. gmem tensor layouts used `order=(0, 1)` (col-major, axis 0
   stride 1) but torch tensors are row-major.  Flipped to
   `order=(1, 0)`.
2. Switched smem layouts to row-major (K stride 1) to match.

After fixes: cute dequant bit-exact vs Python reference
`_dequant_fp4 + _apply_block_scales` chain (max_abs=0.0).

### Step 2 (`b62af89`): gmem→smem copies
Added dedicated `sWpacked` smem region (uint8) so packed FP4 weight
has its own staging area separate from `sB` (bf16 dequant target).
Built two cooperative `make_tiled_copy_tv + CopyUniversalOp` copies
for A (bf16) and W (uint8).

Gotchas resolved:
* `cpasync.commit_group` is at `cute.arch.cp_async_commit_group`,
  not on the `cpasync` module.
* `cp.async` requires 128-bit aligned access; `make_ptr(..., 16)`
  doesn't propagate past `local_tile`.  Fell back to
  `CopyUniversalOp` at element granularity (v1, slow); 128-bit
  widening landed later in v3.1.

### Step 1 (`5373882`): smem allocation
Real CuTe-DSL kernel skeleton with TiledMMA + 5-step body plan
inline.  Allocated sA, sB via `cute.struct.Align[cute.struct.MemRange[...]]`
in `cutlass.utils.SmemAllocator()`.

Three CuTe-DSL gotchas resolved:
1. `cute.make_tiled_mma` needs MLIR context — moved TiledMma /
   layout / SharedStorage construction into `_setup_attributes`
   called from `@cute.jit __call__` (matches dense.py pattern).
2. `@cute.struct` can't parse PEP-563 string annotations — removed
   `from __future__ import annotations`.
3. Inner kernel method must be `@cute.kernel` (returns launchable),
   not `@cute.jit`.  Layouts passed to it typed `cute.Layout`, not
   `cutlass.Constexpr` — otherwise region isolation rejects them
   as "defined outside the region".

## v1 Triton kernel + scaffolding

### CuTe-DSL backend scaffold (`a210586`) → real Triton implementation

### Triton W4A16 dense kernel (`0c35ba3`)
Triton kernel with explicit FP4 nibble decode in `_decode` chain
(nested @triton.jit `def` failed — inlined the decoder).  Bit-exact
on 30 tests across Nano35 shapes × M ∈ {1, 8, 32}.

Tolerance: cos > 0.9999, max_abs ≤ 1% of reference output magnitude
(kernel uses fp32 accumulators, residual is bf16 tile-order round
off).

Perf at M=1 q_proj: cute 149 µs vs TRT-LLM 21 µs → 7× slower.

### Micro kernel skeleton (`55eecfd`)
`DenseGemmW4A16MicroKernel.is_supported` gates m ∈
{1,2,4,8,10,12,16,24,32}, K % 128, N % 16.  Body initially stubbed
through to Python reference, swapped to Triton in `0c35ba3`.

Relaxed K constraint from MoE micro's `K % 512 == 0` to `K % 128 == 0`
because Nano35 hidden=2688 isn't 512-aligned (MoE routes 2688 to
static, not micro; for dense we want 2688 directly).

### Bench harness with TRT-LLM baseline (`a269843`)
Created `benchmarks/benchmark_dense_gemm_w4a16.py`:
* `bench_events` with CUDA events
* TRT-LLM baseline = `torch.ops.trtllm.fp4_quantize + cuda_core_nvfp4_gemm`,
  apples-to-apples (includes activation quant in the timed region).
* FlashInfer `mm_fp4` baseline disabled — it expects B as
  `(K_packed, N)` storage while TRT-LLM emits `(N, K_packed)`.
  Packed-FP4 transpose is nibble-level, deferred.

TRT-LLM lib loaded via `ctypes.CDLL libtensorrt_llm.so + libth_common.so`
to bypass full `tensorrt_llm` Python import (which needs transformers
etc.).

Discovery: `trtllm.fp4_quantize` requires globalScale (not None) when
sfUseUE8M0=False — fixed bench to pass `torch.tensor(1.0, f32)`.

### Python reference + packer (`375bc24`)
`b12x/gemm/w4a16/reference.py`:
* `dense_reference_w4a16(x, w_fp4, w_blockscale, w_alpha)` — dequant FP4
  to bf16 in fp32, then `torch.matmul`, cast to bf16.
* `quantize_dense_weight_to_fp4(w_bf16, weight_scale_2=1.0)` — pack
  bf16 weight to (FP4, swizzled FP8 SF, scalar alpha).

Reuses `_dequant_fp4`, `_apply_block_scales`, `unswizzle_block_scale`
from `b12x.moe.fused.w4a16.reference` and `swizzle_block_scale`,
`pack_grouped_fp4_values`, `_fp4_quantize_values` from
`b12x.cute.fp4`.

CPU-side roundtrip test: `cos > 0.99` for synthetic random weights
(max_abs accumulates with sqrt(K)).
